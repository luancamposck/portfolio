## What Are Server Components?

React Server Components (RSC) are components that **render exclusively on the server**. They never ship JavaScript to the browser, which means they can directly access databases, file systems, and other server-only resources.

In Next.js 16, every component is a Server Component by default. You opt into client rendering with the `"use client"` directive.

## The Mental Model

Think of it as two layers:

```
Server Layer (default)          Client Layer ("use client")
├── Can access DB, fs, env      ├── Can use useState, useEffect
├── Can use async/await          ├── Can use event handlers
├── Zero JS sent to browser      ├── JS shipped to browser
└── Cannot use hooks/events      └── Cannot access server resources
```

The key insight: **Server Components can import and render Client Components, but not the other way around.**

## When to Use Each

### Server Components (default)

Use for anything that doesn't need interactivity:

```tsx title="src/app/blog/[slug]/page.tsx"
import { getPost } from "@/data/blog-posts"

export default async function BlogPost({
  params,
}: {
  params: Promise<{ slug: string }>
}) {
  const { slug } = await params
  const post = await getPost(slug)

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.description}</p>
      <PostContent content={post.content} />
    </article>
  )
}
```

### Client Components

Use only when you need browser APIs, state, or event handlers:

```tsx title="src/components/like-button.tsx"
"use client"

import { useState } from "react"

export function LikeButton({ postId }: { postId: string }) {
  const [liked, setLiked] = useState(false)

  return (
    <button onClick={() => setLiked(!liked)}>
      {liked ? "❤️" : "🤍"} Like
    </button>
  )
}
```

<Callout variant="info">
The "use client" directive marks a **boundary**, not just a single component. Every module imported by a client component also becomes part of the client bundle.
</Callout>

## Common Patterns

### Pattern 1: Server Component with Client Islands

Keep the page as a Server Component and only make interactive pieces client:

```tsx title="src/app/dashboard/page.tsx"
import { getMetrics } from "@/lib/metrics"
import { MetricsChart } from "@/components/metrics-chart" // "use client"
import { DatePicker } from "@/components/date-picker"     // "use client"

export default async function Dashboard() {
  const metrics = await getMetrics()

  return (
    <div>
      <h1>Dashboard</h1>
      <DatePicker />
      {/* Server-fetched data passed as props to client component */}
      <MetricsChart data={metrics} />
    </div>
  )
}
```

### Pattern 2: Composition via Children

Pass Server Components as `children` to avoid making them client components:

```tsx title="src/components/sidebar-layout.tsx"
"use client"

import { useState } from "react"

export function SidebarLayout({ children }: { children: React.ReactNode }) {
  const [open, setOpen] = useState(true)

  return (
    <div className="flex">
      <aside className={open ? "w-64" : "w-0"}>
        <button onClick={() => setOpen(!open)}>Toggle</button>
      </aside>
      <main>{children}</main>
    </div>
  )
}
```

```tsx title="src/app/page.tsx"
import { SidebarLayout } from "@/components/sidebar-layout"
import { ServerContent } from "@/components/server-content"

export default function Page() {
  return (
    <SidebarLayout>
      {/* This stays a Server Component! */}
      <ServerContent />
    </SidebarLayout>
  )
}
```

<Callout variant="warning">
A common mistake is adding "use client" to a parent component when only a small child needs interactivity. This pulls the entire subtree into the client bundle.
</Callout>

## Common Pitfalls

### 1. Importing server-only code in client components

```ts
// This will fail at build time
"use client"
import { db } from "@/lib/database" // Server-only!
```

### 2. Forgetting that params is a Promise in Next.js 16

```ts
// Wrong — params is a Promise now
export default function Page({ params }: { params: { id: string } }) {
  return <div>{params.id}</div> // undefined!
}

// Correct
export default async function Page({
  params,
}: {
  params: Promise<{ id: string }>
}) {
  const { id } = await params
  return <div>{id}</div>
}
```

### 3. Making components client when they don't need to be

Ask yourself: **does this component use `useState`, `useEffect`, `onClick`, or browser APIs?** If no, it should stay a Server Component.

## Performance Impact

Server Components reduce the JavaScript bundle sent to the browser. In a typical Next.js app:

- **Before RSC**: Every component's code ships to the client
- **After RSC**: Only interactive components ship JS

This means faster initial page loads, better Core Web Vitals, and improved SEO — especially on slower connections and devices.

## Conclusion

The rule of thumb is simple: **start with Server Components and only add "use client" when you need interactivity**. Keep client boundaries as small and as deep in the component tree as possible.

This isn't just a Next.js pattern — it's the future of React itself.
