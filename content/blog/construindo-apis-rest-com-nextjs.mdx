## Introdução

O Next.js 16 consolidou os **Route Handlers** como a forma padrão de criar endpoints de API. Diferente das antigas API Routes do diretório `pages/api`, os Route Handlers vivem no App Router e aproveitam toda a infraestrutura de caching e streaming.

Neste post, vamos construir uma API REST completa com **CRUD**, validação de dados e tratamento de erros consistente.

## Anatomia de um Route Handler

Um Route Handler é um arquivo `route.ts` dentro do App Router que exporta funções nomeadas pelos métodos HTTP:

```ts title="src/app/api/posts/route.ts"
import { NextRequest, NextResponse } from "next/server"

export async function GET(request: NextRequest) {
  const posts = await db.post.findMany()
  return NextResponse.json(posts)
}

export async function POST(request: NextRequest) {
  const body = await request.json()
  const post = await db.post.create({ data: body })
  return NextResponse.json(post, { status: 201 })
}
```

<Callout variant="info">
Route Handlers no App Router substituem completamente as API Routes do `pages/api`. Se você está migrando, a mudança principal é que cada método HTTP é uma função exportada separada.
</Callout>

## Validação com Zod

Nunca confie em dados que chegam do cliente. Use **Zod** para validar o body das requisições:

```ts title="src/app/api/posts/route.ts" {4-8,13}
import { z } from "zod"

const createPostSchema = z.object({
  title: z.string().min(3).max(200),
  content: z.string().min(10),
  tags: z.array(z.string()).optional().default([]),
  published: z.boolean().optional().default(false),
})

export async function POST(request: NextRequest) {
  const body = await request.json()
  const result = createPostSchema.safeParse(body)

  if (!result.success) {
    return NextResponse.json(
      { error: "Validation failed", issues: result.error.issues },
      { status: 400 }
    )
  }

  const post = await db.post.create({ data: result.data })
  return NextResponse.json(post, { status: 201 })
}
```

O `safeParse` retorna um resultado tipado sem lançar exceção — perfeito para APIs onde queremos retornar erros estruturados.

## Error Handling Consistente

Crie um helper para padronizar respostas de erro:

```ts title="src/lib/api-error.ts"
export class ApiError extends Error {
  constructor(
    public statusCode: number,
    message: string,
    public details?: unknown
  ) {
    super(message)
  }
}

export function handleApiError(error: unknown) {
  if (error instanceof ApiError) {
    return NextResponse.json(
      { error: error.message, details: error.details },
      { status: error.statusCode }
    )
  }

  console.error("Unexpected error:", error)
  return NextResponse.json(
    { error: "Internal server error" },
    { status: 500 }
  )
}
```

## Route Params Dinâmicos

Para rotas como `/api/posts/[id]`, o Next.js 16 passa `params` como uma **Promise**:

```ts title="src/app/api/posts/[id]/route.ts"
export async function GET(
  request: NextRequest,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params

  const post = await db.post.findUnique({ where: { id } })

  if (!post) {
    return NextResponse.json(
      { error: "Post not found" },
      { status: 404 }
    )
  }

  return NextResponse.json(post)
}
```

<Callout variant="warning">
No Next.js 16, `params` é sempre uma Promise. Esquecer o `await` é um dos erros mais comuns na migração.
</Callout>

## Caching e Revalidação

Route Handlers GET sem usar `request` são automaticamente cacheados. Para controlar o cache:

```ts
// Revalidar a cada 60 segundos
export const revalidate = 60

// Ou desabilitar cache completamente
export const dynamic = "force-dynamic"
```

## Conclusão

Route Handlers no Next.js 16 oferecem uma API limpa e type-safe para construir backends. Combinados com Zod para validação e um error handler consistente, você tem tudo para criar APIs robustas sem sair do ecossistema Next.js.
