## Codebase Patterns
- Use `next/font/local` instead of `next/font/google` — Google Fonts cannot be fetched at build time in this environment. Font files go in `public/fonts/`.
- Font CSS variable chain: layout.tsx sets `--font-varela-round` on `<html>` via `varelaRound.variable` class → globals.css `@theme inline` maps `--font-sans: var(--font-varela-round)` → `html` applies `@apply font-sans`
- Geist font files already exist in `public/fonts/` (geist-latin.woff2, geist-mono-latin.woff2)
- This codebase uses tabs for indentation — when using Edit tool, match with unique string fragments rather than leading whitespace
- Biome is the linter/formatter — run `npx biome check --fix src/` after edits
- Responsive mobile-first pattern: `gap-3 sm:gap-6`, `p-4 sm:p-6`, `text-2xl sm:text-3xl` etc.
- `buttonVariants()` outputs static classes — cannot be made responsive; use default size or custom responsive className
- Next.js 16 renamed `middleware.ts` to `proxy.ts` — export function `proxy` (not `middleware`), file goes at `src/proxy.ts`
- next-intl v4 setup: `src/i18n/routing.ts` (defineRouting), `src/i18n/request.ts` (getRequestConfig), `src/i18n/navigation.ts` (createNavigation)
- next-intl plugin chain in next.config.ts: `withMDX(withNextIntl(nextConfig))` — withNextIntl wraps inner, withMDX wraps outer
- Translation files in `messages/{locale}.json` — loaded dynamically in `src/i18n/request.ts`
- Root layout (`src/app/layout.tsx`) handles fonts + html/body only; `[locale]/layout.tsx` handles ThemeProvider, Navbar, Footer, NextIntlClientProvider
- `setRequestLocale(locale)` must be called in `[locale]/layout.tsx` for static rendering to work
- `params` is always `Promise<{...}>` in Next.js 16 — always `await params` in pages and layouts
- Translation namespaces mirror component names: `useTranslations('hero')`, `useTranslations('navbar')`, `getTranslations('projectsPage')` etc.
- ICU message format for dynamic values: `t('copyright', { year })`, `t('imageAlt', { title, index })`
- All internal links use `Link` from `@/i18n/navigation` (not `next/link`) — auto-prepends locale
- `usePathname()` from `@/i18n/navigation` returns path without locale prefix — active state comparisons use clean paths like `/projects`
- Server components needing translations must be `async` and use `getTranslations()` from `next-intl/server`
- Pages with translated metadata use `generateMetadata()` (not static `export const metadata`) + `setRequestLocale(locale)` inside
- Turbopack requires string-based rehype/remark plugin names in `createMDX` options — function references and non-serializable options (callbacks) are not supported
- rehype-pretty-code CSS: use `data-*` attributes (`data-highlighted-line`, `data-highlighted-chars`, `data-line-numbers`, `data-rehype-pretty-code-title`) for styling — no need for callback-based class names
- `buttonVariants()` from `button-variants.ts` gets treated as client code by Turbopack because `button.tsx` ("use client") imports it — extract into client components when needed in server components
- Root `not-found.tsx` (outside `[locale]`) has no i18n context — use plain strings, not `useTranslations()`
- Dual theme config for rehype-pretty-code: `{ dark: "github-dark-default", light: "github-light-default" }` — CSS switches via `var(--shiki-light)` / `var(--shiki-dark)` with `.dark` selector
- Custom MDX components live in `src/shared/components/mdx/` and are registered in `src/mdx-components.tsx` — available globally in all MDX content without imports
- The `pre` key in `useMDXComponents()` overrides rehype-pretty-code's `<pre>` output — used for CodeBlock with copy button
- For raw code string highlighting (non-MDX): use `shiki.codeToHtml()` server-side with same theme config as rehype-pretty-code
- For markdown string → HTML (non-MDX): `unified().use(remarkParse).use(remarkRehype)` + `hast-util-to-html` — no extra installs needed
- `useSearchParams()` requires `<Suspense>` wrapper for SSG — otherwise build fails
- Blog posts are MDX files in `content/blog/` read via `fs` + `gray-matter` — the `BlogPost` type and utilities are in `src/data/blog-posts.ts`
- `getAllPosts()` returns published posts sorted by date desc; `getPostBySlug()` returns any post (including drafts)
- Client components ("use client") CANNOT import modules using `node:fs` — data from `blog-posts.ts` must be fetched in server components and passed as props
- `next-mdx-remote/rsc` `MDXRemote` renders MDX strings server-side — use for blog post content with rehype/remark plugins (function references OK, not limited by Turbopack)
